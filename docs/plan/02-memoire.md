## 1. Objectif & ce que la couche prouve

Bâtir la **mémoire** sur le socle : un **fichier de vérité unique** (WAL, écrivain unique = orchestrateur, hérité `00`), une **recherche hybride multi-corpus**, une **écriture à trois portes un seul stylo**, une **consolidation** micro (le jour) et deep (la nuit), une **injection** bornée, un **effacement souverain** réel, deux étages étanches (vécu / lu) réunis à la lecture.

**Ce qu'elle doit prouver** : le **vécu brut est immuable** (sas seul passage, trace sans contenu) · la **recherche hybride** tient (exact + sens + RRF + filtres dans les jambes + requête datée + jamais bloquante + citabilité) · « **c'est noté** » **vrai à la seconde** · **la nuit ne dépend jamais du micro**, la **deep exige le vrai cerveau en entier** · l'**effacement souverain** efface réellement (cascade + invalidation du fil Claude) · l'**identité est inviolable par la mémoire** (contrat 2→3 ; **embeddings local-only**).

**Ce que ce plan ne couvre PAS** : tables persona/lien/cliquet/journal du devenir + usage affect + cadence humeur → `03` (le plan `02` pose **substrat, patron, slots**) · initiatives + cible `task` + politique tablée/tiers → `04` · ladder résilience/coût + jeton OAuth + sauvegarde hors-machine → `05` · composition finale du prompt → `99` (le plan `02` compose **le bloc mémoire seul**) · socle → `00` (acquis).

### M0 — Schéma & immutabilité
- **But** : toutes les tables de la couche 2 dans le WAL unique ; le vécu immuable par construction ; un seul sas de suppression.
- **Dépend de** : socle T1 (WAL + écrivain unique F2).
- **Fichiers** : `db/schema-02.sql`, `src/orchestrator/db/`.
- **Contenu — construit pièce par pièce** :

  **▸ Pièce 1 — l'épisodique immuable + le verrou** *(le sol de l'anti-dérive A18 : « réécrire depuis la source » exige une source intouchable)*
  - **`sessions` — enveloppe MUTABLE** (donc **pas** de trigger insert-only) : `id` (identité **logique stable** de la conversation) · `mode` (**CHECK** : conversation/dictée/tablée/rêverie — **discriminant de pipeline** : dictée & rêverie = enveloppe-seule ; tablée = tiers hors `conversations`) · `retention_policy` (→ `04`) · `summary` (**NULL = « à résumer »**, définition par l'état T20) · `started_at` · `last_active`. **Couture socle — M-1 réorienté (croisé conv 16, §7)** : le **fil Claude vit dans `session_state`** (socle, **propriétaire** — `05` §4.1 T3 / `02` §2.4 T1) ; **`sessions` ne porte PAS `claude_session_id`** (fil **remplaçable en place** à la rotation/invalidation ; continuité par `conversations`) ; c'est `session_state.current_session_id`→`sessions.id` qui **pointe la ligne courante**, MAJ **même transaction** (T14).
  - **`conversations` — IMMUABLE** (insert-only, sas seul) : **`id INTEGER PRIMARY KEY AUTOINCREMENT`** *(ids **jamais réutilisés** — requis par la dérivation « source effacée » de `plan/03` P10/R1 : sans lui, SQLite recycle l'`id` d'un tour effacé au prochain INSERT et une provenance identitaire citerait **un autre tour bien réel** — méprise silencieuse ; retouche remontée par le croisé de `plan/03`, conv 17, §7)* · `session_id` · `role` (**CHECK** user/assistant/system) · `speaker` (Yohann/Sophia/… — politique d'écriture `04` : en pratique Yohann/Sophia, les tiers → `tablee_buffer`) · `content` (**texte, jamais l'audio**) · `surface` (**redéfini**, cf. §7) · `created_at`. **Inserts en `synchronous=NORMAL`** (trafic fréquent, comme `facts` T21 — dernier tour perdable sur coupure secteur, assumé).
  - **`turn_signals` — table SÉPARÉE, rétention bornée** (donc **pas** insert-only) : `conversation_id` · `reason` (fin de tour) · barge-in (survenu/position) · affect (valence/énergie/confiance — **nullable, OFF défaut, verrou Yohann** — `01` §2.4) · **valeurs du tag d'humeur** (deltas/drapeau — **colonnes déclarées ici, écrites par `03`** §2.2, AT6) · `captured_at`. **Séparée pour deux raisons** : OFF-safe + rétention (exception nommée à « le système ne supprime jamais », T11/F6). **Invariant gravé (m-4/M-2 croisé conv 16, non « à calibrer ») : `rétention ≥ borne du backlog de consolidation`** — `turn_signals` est relue par la réécriture nocturne du lien (« texture affect du jour », `03` §4.5) ; purger avant que la nuit (rattrapage multi-jours) l'ait lue = **perte silencieuse de l'affect**. C'est un **couplage structurel** (la valeur de rétention est *dérivée* de la borne backlog, pas indépendante), à asserter au test.
  - **Le verrou (triggers + sas)** : triggers `BEFORE UPDATE/DELETE` → `RAISE(ABORT)` avec une clause **fail-closed** : `WHEN NOT EXISTS (SELECT 1 FROM erase_gate WHERE open=1)` — protégé **par défaut**, ouvert **seulement** si une ligne `open=1` existe (**correctif B1** : `WHEN (…)=0` était *fail-open* — garde vide/`NULL` → `NULL=0` non-vrai → tout passait, sur `conversations`/`chronicle` **et** `identity_core`) ; **`erase_gate`** *(nom d'implémentation ; `02` §2.4 dit « table-garde »)* = **garde mono-ligne** (`id INTEGER PRIMARY KEY CHECK(id=0)`, `open NOT NULL DEFAULT 0`), **seedée à l'init** (`INSERT id=0, open=0` — sinon le sas M8 fait `UPDATE` sur **0 ligne** et n'ouvre jamais : effacement cassé — **m-1 croisé conv 16**) et **protégée contre DELETE** (trigger — **pour garder le singleton présent** que le sas puisse basculer ; sa suppression **FERME** le verrou, fail-closed, elle **ne rouvre pas** — *correctif du motif inverse « la vider rouvrirait tout » = raisonnement fail-open pré-B1, contredisait le test ; **B-β croisé conv 16***). Le **sas** (effacement M8 + soupape du gardien `03`, tracés) fait `open=1` → DELETE → `open=0` **dans une seule transaction, sur la connexion d'écriture de l'orchestrateur**. **Sûr par construction** : l'**atomicité + l'isolation WAL** rendent une garde `open=1` non commitée invisible et la referment au rollback (*jamais persistée ouverte*) ; **F2** (aucun writer sidecar/MCP) **complète** sans porter seul la garantie — précondition : open/DELETE/close **dans une transaction, connexion d'écriture**. **Défense en profondeur (B-α croisé conv 16)** : au **boot**, l'orchestrateur **force `erase_gate.open=0` inconditionnellement + assertion** (filet si un `open=1` était malgré tout persisté par une implémentation hors-transaction — sans ça, le verrou déverrouillerait **tout** en silence au réveil). **Couture socle → `plan/00` T5 phase 1 (ré-audit).** **Patron réutilisable** (générateur de triggers sur la même `erase_gate`) pour les 6 tables identitaires de `03` (contrat 2→3). **S'applique à `conversations` + `chronicle` (+ tables `03`) seulement** — pas à `sessions` (mutable), `turn_signals` (rétention), `facts` (write-once *de contenu* = mécanisme distinct, pièce 3).

  **▸ Pièce 2 — empreintes** (`imprints`) *(le substrat des « empreintes du jour » A18/A19 : le jour note, la nuit range ; `02` §3.2 délègue à `03` la **sémantique du tri** — on pose le **slot, pas la politique**)*
  - **`imprints` — enveloppe MUTABLE, hors verrou** (`consumed` réécrit par la nuit → **pas** insert-only, **pas** sous `erase_gate`) : `id` · **ancrage XOR** `conversation_id` (→`conversations.id`, un tour vécu) **⊕** `session_id` (→`sessions.id`, invocation **sans tours** : rêverie = session enveloppe-seule, `03` T3) — exactement l'un des deux (**`CHECK ((conversation_id IS NULL) + (session_id IS NULL) = 1)`**) · `nature` (**slot**, TEXT sans CHECK — vocabulaire/sens = `03`) · `priority` (**slot** — l'« empreinte **prioritaire** » de `memory_store` §2.3 + de la révélation-en-conversation §5 ; le tri = `03`) · `noted_by` (**CHECK fermé** `micro`/`outil`/`couche3`, les trois plumes de `02` §3.2) · `consumed` (bool, défaut 0 — **mis à 1 par la nuit** quand la deep la consomme, §4.4 étage 2 ; **mutable → aucun trigger**) · `created_at`.
  - **Ancrage = deux FK typées + XOR** (tranché) : intégrité FK réelle sur les deux cibles (tables en base) + « exactement un ancrage » **invariant de base**. *Écarté* : un couple polymorphe `(anchor_kind, anchor_id)` (perd la FK ; le polymorphe ne se justifie que pour des cibles hétérogènes — cas de `fact_sources.source_id`, pas ici) ; un tour-sentinelle pour la rêverie (**fausse ligne** dans `conversations` immuable — contraire à « sans faux passé »).
  - **Cascade d'effacement = explicite (M8), jamais `ON DELETE CASCADE`** (tracé §7) : les empreintes ancrées sur un tour effacé sont supprimées **dans la transaction M8**, comptées par la trace sans contenu (§2.4/T18) — une cascade implicite rendrait le compte opaque. M0 garantit seulement que le schéma le permet (FK déclarées, ligne supprimable par l'orchestrateur).

  **▸ Pièce 3 — sémantique** (`facts` · `fact_sources` · `fact_relations`) *(le write-once du **contenu** — mécanisme distinct du verrou de la pièce 1, T7)*
  - **`facts`** : `id` · `content` (**langage naturel concis** — ce qui est embeddé, A11) · `category` (**vocab fermé** `CHECK` : personne/préférence/relation/projet/quotidien/monde/système — **extension = migration de schéma, jamais la nuit** : le CHECK *est* l'enforcement de « jamais par la nuit ») · `status` (**CHECK** `PROVISIONAL`/`ACTIVE`/`SUPERSEDED`/`REJECTED`) · `confidence` · `importance` · `support_count` (**cache** de la cardinalité de `fact_sources`) · `valid_from`/`valid_to` (**temps du monde**) · `created_at` (**temps de la croyance**).
  - **Write-once du contenu (T7)** — trigger `BEFORE UPDATE` **distinct du verrou** : refuse la modification des colonnes **d'identité du fait** (`content`, `category`, `valid_from`, `created_at`) via `RAISE(ABORT)` ; **autorise** les seules **métadonnées mutables** nommées par T7 (`status`, `valid_to`, `support_count`, `importance`, `confidence`). **Comparaison NULL-safe obligatoire (b-m2 croisé conv 16 — le piège exact de B1)** : tester `NEW.x IS NOT OLD.x`, **jamais** `NEW.x <> OLD.x` (qui rend `NULL` — non-vrai — sur une transition `NULL↔valeur`, ex. `valid_from`, et **laisse passer** la modif interdite) ; à prouver par un test de transition impliquant `NULL`. *Pourquoi ce mécanisme et pas l'insert-only+gate de la pièce 1* : un fait **doit** muter ses métadonnées (ratification, supersession) et être **supprimé** par l'effacement souverain — il n'est donc ni immuable ni sous `erase_gate` ; seul son **contenu embeddé** est scellé (sinon : vecteur périmé invisible à la file §2.2 + FTS jamais rafraîchi = corruption d'index silencieuse).
  - **`fact_sources`** : `fact_id` · `source_kind` (**CHECK fermé** `tour`/`consolidation`/`témoignage`/`outil`/**`réactivation`** — cette 5ᵉ valeur ajoutée conv 16 pour la provenance citable d'un fait réactivé par effacement, #E ; extension de vocab par décision explicite, tracée §7) · `source_id` (**polymorphe** — cible selon `source_kind` : un tour, un run, un doc-témoignage, un appel outil, **ou une `erasures`** ; pas de FK stricte) · `observed_at` · `source_erased` (marqueur « source effacée », §2.4/T4 — la provenance reste honnête quand le tour pointé est effacé, la chaîne ne casse pas en silence). **Index `(source_kind, source_id)`** (b-m6-index croisé conv 16) : le **balayage inverse** de M8 (retrouver les sources pendantes vers un tour effacé pour les marquer) doit être **performant et exhaustif** — sans index, ni l'un ni l'autre.
  - **`fact_relations`** : `from_fact_id` · `relation` (**CHECK fermé** `SUPERSEDES`/`CONTRADICTS`/`DERIVES_FROM`) · `to_fact_id` · `basis` (→`conversations.id`, le tour de la correction) · `created_at` — **`CHECK (relation <> 'SUPERSEDES' OR basis IS NOT NULL)`** : `SUPERSEDES` refuse sans base (§4.4, critère 13) ; `CONTRADICTS` la tolère (les deux faits gardés, tension exposée).
  - **`support_count` maintenu par l'orchestrateur, même transaction** que l'écriture de `fact_sources` (tranché — philosophie T22 « posés dans la même transaction », single-writer ; *écarté* : des triggers-compteurs cachés, moins auditables). **`status=SUPERSEDED` ⇔ relation `SUPERSEDES` entrante (T22)** : posés dans la **même transaction** par la nuit — invariant tenu par l'écrivain (non exprimable en CHECK inter-tables), vérifié au test.
  - **Hors verrou, contenu scellé, supprimable par le seul M8** : `facts`/`fact_sources`/`fact_relations` ne sont **pas** sous `erase_gate`. Le **système ne supprime jamais** un fait (il marque `SUPERSEDED`/`REJECTED` — discipline de code de la nuit) ; **le seul DELETE = l'effacement souverain M8** (§2.4). La fusion de doublons (nuit) **crée un nouveau fait** (`DERIVES_FROM`) + marque les anciens `SUPERSEDED` — **jamais** de réécriture en place (T7).

  **▸ Pièce 4 — artefacts + chronique** (`memory_artifacts` · `chronicle`)
  - **`memory_artifacts`** (patron réutilisé par `03` pour le miroir-lien — contrat 2→3) : `name` (`user_model`, …) · `version` · `content` · `written_by` (run de consolidation) · `created_at` · **`expurged_at` (nullable — marqueur « expurgée le … », patron T3, parité `chronicle` ; AF-6, audit Fable : la cascade M8/T14 expurge des versions, le marqueur n'avait nulle part où vivre)** — **`UNIQUE(name, version)`** (b-m3 croisé conv 16 : « le courant = la dernière » + rotation « garder N » sont ambigus si deux `(user_model, 5)` coexistent — parité avec `chronicle.day UNIQUE`) ; **historique borné à N versions**, le courant = la dernière. **`user_model` vit dans le WAL** (garantie #4 du socle : état durable centralisé → snapshots/récupération uniformes ; supersession de forme du `user_model.md` libre du cahier, `02` §7).
  - **Une version est immuable, la table n'est pas sous `erase_gate`** (tranché) : trigger `BEFORE UPDATE` → `RAISE(ABORT)` (une version ne se réécrit jamais — traçabilité A18 ; réécrire = **nouvelle** version) ; **INSERT libre** (version++) ; **DELETE autorisé à l'orchestrateur** pour la **rotation bornée** (exception nommée §5/T11) **et** l'**expurgation** par effacement souverain (§2.4/T4, patron T3). *Distinction subtile tracée §7* : `memory_artifacts` **n'est pas** verrouillée comme `chronicle` — la rotation des N versions est un effacement *système* nommé et borné, pas un contournement du sas.
  - **Le portrait 4 strates + 2 règles = critères d'écriture dans le prompt de consolidation versionné** (`02` §3.4/§4.4), **pas des colonnes** : le schéma stocke `name`/`version`/`content`. **Les 4 strates sont une structure *gravée*, pas de la calibration (A-3 croisé conv 16)** : **1 Essence** (caractère, valeurs — années) · **2 Goûts & manières durables** (mois/années) · **3 Contexte de vie** (semaines) · **4 Préférences opérationnelles** (immédiat) ; **Règle 1 — double provenance visible** (« tu m'as dit » ≠ « j'ai observé », E1) et **Règle 2 — l'essence se gagne lentement** sont des **exigences fixes du prompt**, testables (M5, § tests). L'amorçage v0 (héritage témoignages) écrit une **première version** de provenance `témoignage` (tâche pré-boot / M5).
  - **`chronicle` — IMMUABLE (verrou pièce 1) + `day` UNIQUE** : `day` (**`UNIQUE`** — jamais deux entrées pour un même jour, critère 10) · `content` (le jour vécu, raconté) · `written_at` · `written_by` (run) · `expurged_at` (nullable — marqueur « expurgée le … », §2.4/T3). **Sous le générateur de triggers de la pièce 1** (insert-only + `erase_gate`) : UPDATE/DELETE refusés hors sas ; l'expurgation = **DELETE + réinsertion expurgée-marquée** dans le sas (T3, **jamais d'UPDATE**, jamais de faux journal). **Deux dates** : `day` couvert ≠ `written_at` (un rattrapage multi-jours écrit des entrées datées de *leurs* jours, marquées écrites plus tard) — heure de coupure du « jour » (alignée A33) = calibration (§6).

  **▸ Pièce 5 — knowledge** (`knowledge_docs` · `knowledge_chunks`) *(l'étage (b) — ingéré, immuable, **jamais consolidé** ; la nuit ne le lit ni ne le touche)*
  - **`knowledge_docs`** : `id` · `name` · `hash` · `version` · `status` (**CHECK** `ACTIVE`/`REPLACED`/`ARCHIVED` — **le seul champ de cycle de vie mutable**) · `category` (**TEXT ouvert**, dont `témoignage` ; tranché — `02` « dont témoignage » n'énumère **pas** de vocabulaire fermé, à l'inverse de `facts.category` : on ne referme pas ce que la source laisse ouvert) · `ingested_at` — l'inventaire (« qu'as-tu lu de moi ? » a une vraie réponse). **Index partiel `UNIQUE(name) WHERE status='ACTIVE'`** (b-m3 croisé conv 16 : « une seule version servie » — jamais deux docs `ACTIVE` de même `name`).
  - **`knowledge_chunks`** : `doc_id` (→`knowledge_docs.id`) · `seq` · `content` — **contenu write-once** (trigger `BEFORE UPDATE` → `RAISE`, comme `facts.content` ; un fichier modifié = **nouvelle version** de doc, jamais une réécriture de morceau → pas de « morceaux fantômes »). Découpe (titres/paragraphes, chevauchement) = calibration (§6) ; la **mécanique** d'ingestion/versionnage = **M7**, M0 ne pose que le schéma qui la porte. **`chunks` NE porte PAS de statut** (il vit sur `knowledge_docs`) : les chunks d'un doc `REPLACED`/`ARCHIVED` **restent en base** (inventaire) **et indexés** → « plus jamais servi » n'est PAS gratuit. **Enforcement (B-θ croisé conv 16)** : la jambe knowledge de M2 **filtre par jointure `chunk → knowledge_docs WHERE status='ACTIVE'`, poussée DANS les deux jambes** (FTS + vec, T5) — sinon la recherche sert les **morceaux fantômes** que M7 jure d'éviter.
  - **Hors verrou, mais la nuit n'y touche jamais** (invariant, critère 14) : `knowledge_*` n'est pas sous `erase_gate` (statut mutable + effacement dur §2.4 par l'orchestrateur) ; la garantie « **zéro écriture nocturne dans `knowledge`** » est tenue par le **code de consolidation** (la nuit ne référence pas ces tables) + **audit** — prouvée à M5/M7, notée ici.

  **▸ Pièce 6 — index dérivés + journaux** (`*_fts` · `vec_*` · `embed_space_meta` · `consolidation_runs` · `erasures`)
  - **FTS5 (contenu externe), triggers `INSERT`+`DELETE` seulement — jamais d'UPDATE (T7)** : `facts_fts` · `conversations_fts` · `sessions_fts` (résumés, T6) · `chronicle_fts` · `knowledge_fts`.     2. **Migration corpus par corpus, par priorité** (`facts` d'abord, puis `chronicle`/`sessions`/**`introspection`** (`03`, AT7)/`knowledge` — **les vécus avant le lu**) : reconstruction en **table ombre dotée de sa PROPRE méta cible** (B-ζ : le garde d'écriture M1 vérifie contre la méta de la table écrite → l'ombre accepte le nouvel espace pendant que l'actif reste l'ancien) — tâche de fond gouvernée, **resumable** (unités+curseur). **Pendant la fenêtre de migration d'un corpus, TOUTES ses écritures d'embed vont dans l'ombre** (rattrapage de l'ancien **et** lignes neuves créées en vol, ex. « retiens que X ») — le garde accepte (méta cible) → **aucune tempête de refus ni d'alerte T17** (#1 croisé conv 16) ; l'actif reste gelé, **lu en FTS-seul** (B-η). **Bascule atomique** = **une transaction (réclamation comprise — B-A/#G 3ᵉ croisé conv 16)** : `DROP TABLE vec_<corpus>` (ancien — **réclame le disque**) → `RENAME vec_<corpus>__mig → vec_<corpus>` → dans `embed_space_meta` **`DELETE` la ligne de l'ancien espace** (sinon **fuite disque** + **collision `UNIQUE` au retour à un modèle antérieur** — la migration-retour avorterait) **puis** **`active=1`** sur la ligne cible (l'ordre passe par un état « zéro active » légal, imposé par le partial-unique M0 — #2/#6) → **reset des marqueurs `embed_failures` du corpus** (#F : nouveau modèle = nouvelle chance de racheter une poison-row) → le corpus retrouve sa jambe sémantique. *(`vec_initiatives` = **flushé, pas migré** — `04`.)*
    3. Les lignes créées **pendant** la reconstruction sont donc **déjà dans l'ombre** ; toute résiduelle est rattrapée **après bascule** par la règle générale (« à embedder = lignes sans vecteur », M1 — fidèle à `technique/02` §4.8-3 « **après bascule** »).
  - **Corpus pas-encore-migré, écritures différées (#A-2 4ᵉ croisé conv 16)** : lors d'un changement de modèle, **tous** les corpus sont « en attente » ; un corpus **dont la migration n'a pas commencé** (pas d'ombre, méta active = ancien espace, sidecar déjà au nouveau) **ne tente PAS d'embed** — ses lignes neuves restent **FTS-seul** (symétrique au garde de lecture B-η) jusqu'à ce que **son** tour de migration vienne (elles rejoignent alors son ombre). Sans ça, chaque ligne neuve d'un corpus en attente serait embeddée au nouveau modèle, **refusée** par le garde (méta ancienne) → **tempête d'alertes T17** — le « refus d'espace transitoire attendu » (#5) exclut donc **aussi l'alerte T17**, pas seulement le dead-letter.
  - **Reprise après crash — setup IDEMPOTENT (#A-3 4ᵉ croisé conv 16)** : le montage d'une migration (ligne cible `active=0` + table ombre) est **idempotent** (`CREATE TABLE IF NOT EXISTS vec_<corpus>__mig` ; INSERT de la méta cible **seulement si absente**) → au reboot mid-migration, le setup **reprend l'ombre existante** (base=file la remplit), ne la recrée pas (sinon collision `UNIQUE`/`already exists` → migration bloquée, ombre orpheline). **Nettoyage d'une migration avortée** : si le modèle configuré ≠ la cible d'une ombre subsistante → **DROP ombre + DELETE méta cible** avant de démarrer la nouvelle (la contrainte « ≤1 cible par corpus », M0 p6, garantit qu'il n'y en a jamais deux).
  - **Jamais deux espaces mélangés dans un même KNN.** Le **jeu-témoin du dimanche** (greffé sur le health-check A18/A37, socle) re-embedde un petit ensemble **par corpus** et vérifie l'**auto-retrouvabilité KNN** — une mémoire sémantique qui se dégrade *en silence* est le pire mode de panne, indétectable sans ça (et tout refus du garde alerte immédiatement, T17, M1).
- **Fait quand (U-M9)** : changement de modèle → **dégradation globale annoncée** → tous corpus **FTS-seul** → **migration ombre par priorité** (vécus avant lu) → **bascule atomique** (`embed_space_meta.active`) ; **jamais deux espaces dans un même KNN** ; lignes créées pendant → rattrapées (base = file) ; **jeu-témoin du dimanche par corpus** (auto-retrouvabilité KNN) ; `vec_initiatives` flushé (`04`), pas migré. *(U-M9, I-M9.)*
